Detailed Findings
Codebase Health & Maintainability
70
C
Well-organized codebase with zero TODO/FIXME markers and a 1:1 test coverage ratio demonstrating strong discipline. However, file-size outliers (ApiDocs.jsx at 842 LOC, schemas.py at 473 LOC) and sparse error handling across 1,621 backend LOC create maintainability friction. The monolithic MCP surface duplicated across three endpoints with no compile-time safety compounds technical debt.
Findings
High
ApiDocs.jsx is an 842-LOC monolith with embedded static JSON
The entire API documentation page is a single component with hardcoded endpoint descriptions and example JSON payloads. Any API schema change requires manual updates across multiple string blocks with no single source of truth.
High
mcp_server.py bypasses REST layer — divergent call paths
The embedded MCP server calls service functions and opens SQLAlchemy sessions directly, bypassing the REST router. Two independent implementations of every business operation exist with no shared validation or middleware coverage.
Medium
schemas.py consolidates 473 LOC of Pydantic models in a single file
Request and response schemas for all resources coexist in one file. Any new resource type compounds the file further, degrading IDE navigation and code review.
Medium
Sparse error handling — only 20 try/except for 1,621 backend LOC
Database errors propagate as unhandled 500 responses with full stack traces, potentially exposing internal details to clients.
Medium
Triple-duplicated API surface — 5-component blast radius per schema change
Every API capability independently implemented in REST router, embedded FastMCP server, and both standalone MCP client packages. The packages have already drifted.
Low
No clear separation of concerns between MCP and REST layers
The embedded MCP server diverges from REST patterns, making it harder to maintain consistent behavior across API surfaces.
Recommendations
  • Auto-generate ApiDocs.jsx from the OpenAPI spec at build time. Split each accordion section into its own sub-component.
  • Split schemas.py into domain modules: schemas/prompts.py, schemas/agents.py, schemas/tags.py, schemas/executions.py.
  • Add a global FastAPI exception handler for SQLAlchemyError (503), IntegrityError (409), and NoResultFound (404) with sanitized messages.
  • Route MCP tool calls through the REST API internally to eliminate divergence risk and gain consistent middleware coverage.
  • Generate standalone MCP client stubs from the OpenAPI spec at build time. Add a CI test asserting package tool schemas match the OpenAPI definition.
  • Establish a 400-LOC per-file ceiling in linting rules. Flag files exceeding this in PR checks to prevent future monoliths.
Security & Compliance
72
C
Solid foundational security practices including strict version pinning and permission-minimized CI/CD workflows. Critical gaps exist: SQLite as default database with no encrypted secrets management, permissive CORS configuration, missing HTTPS enforcement, and absence of dependency scanning and secret detection controls.
Findings
Critical
Default SQLite Database in Production
Helm values and database configuration default to SQLite even in production. No encryption at rest, multi-instance consistency, or backup procedures.
Critical
No Secrets Management or Rotation
Credentials (DATABASE_URL, API keys) stored in environment variables via Helm values without encryption, rotation policies, or secret scanning. No Vault or Secrets Manager integration.
High
Permissive CORS — wildcard methods, headers, and credentials enabled
CORS middleware allows all methods ['*'] and all headers ['*'] with credentials enabled. Cross-origin request hijacking and token leakage risks.
High
Missing Dependency Scanning and SCA
CI/CD pipeline contains no SCA scanning, SBOM generation, or CVE monitoring for Python or Node dependencies. No container image scanning.
High
Unencrypted HTTP by Default
Dockerfile and Helm configuration expose services over plain HTTP without enforcing HTTPS, TLS termination, or certificate management.
Medium
Unauthenticated OpenAPI Docs Exposure
/api/docs, /api/redoc, and /api/openapi.json enabled by default in production with no authentication or rate limiting, enabling reconnaissance.
Medium
Component Rendering Injection Risk
Prompt content supports variable substitution and component references via regex-based parsing. Insufficient sanitization of rendered outputs could enable prompt injection.
Medium
Non-Restrictive Database Access Patterns
SQLAlchemy ORM queries use ilike() filters without parameterization guards. SQL injection mitigation relies on ORM defaults rather than enforced patterns.
Low
Missing Security Headers and CSP
No X-Frame-Options, X-Content-Type-Options, or Content-Security-Policy in middleware. Frontend served without CSP to prevent inline script injection.
Low
Incomplete Audit Logging
No audit trail for sensitive operations (prompt creation, deletion, execution recording). Critical for SOC 2, HIPAA, and incident investigation.
Low
Tag key casing — Project not activated as Cost Allocation Tag
All resources receive Project = open-prompt-manager via Terraform default_tags. Key is uppercase Project — Datadog normalises correctly, but AWS Cost Explorer requires activating Project as a Cost Allocation Tag in AWS Billing for project-level cost reports to work.
Recommendations
  • Implement mandatory PostgreSQL for production with encryption at rest, automated backups, and point-in-time recovery configured in Helm and Terraform.
  • Integrate HashiCorp Vault or AWS Secrets Manager for credential rotation with 30–90 day rotation SLA.
  • Add OWASP Dependency-Check or Snyk SCA scanning to CI/CD with fail-on-high/critical CVE policy; generate SBOM artifacts per release.
  • Restrict CORS to explicit origin allowlist from environment config; disable credentials for cross-origin requests unless strictly required.
  • Enforce HTTPS/TLS with cert-manager in Kubernetes; redirect HTTP to HTTPS at ingress; require TLS 1.3 minimum.
  • Disable OpenAPI docs in production via environment flag; require API key or OAuth2 protection for schema introspection.
  • Add Trivy container image scanning to CI/CD; fail on critical/high CVEs in base images.
  • Implement structured audit logging middleware for all API mutations; integrate with centralized logging (ELK, Datadog) for compliance.
  • Create SECURITY.md with threat model, responsible disclosure policy, and SLA for patching critical/high vulnerabilities (7–14 days).
  • Add DAST (OWASP ZAP) scanning in nightly CI jobs to detect runtime vulnerabilities.

  • Implement circuit breaker and retry logic for MCP session manager and database access using tenacity; add exponential backoff with jitter.
  • Replace SQLite with PostgreSQL in production; add connection pooling (PgBouncer), read replicas, and automated backups with point-in-time recovery.
  • Define and publish SLOs: 99.5% availability, p99 latency < 500ms, error rate < 0.1%; create SLI dashboards and error-budget-based release gates.
  • Add FastAPI global exception handler middleware; implement health check aggregation (database, MCP server) with fail-open fallbacks.
  • Implement graceful shutdown with PreStop hooks and request drain timeouts in Kubernetes manifests.
  • Add per-endpoint rate limiting using slowapi or FastAPI-limiter with adaptive load shedding.
  • Run quarterly game days: test database failure scenarios, network latency injection, and partial service degradation; measure RTO and document runbooks.

However, accessibility is severely underdeveloped (only 4 ARIA attributes across 4,700 LOC), input validation has no client-side component mapping, and the absence of error boundaries and toast notifications means users encounter blank-screen crashes and lose context on quick sequential operations.
Findings
Critical
No React ErrorBoundary — blank-screen crashes cascade to users
When any top-level route component throws, there is no error boundary to catch and display graceful fallback UI. Users see a blank page with no recovery path and no error logging occurs.
High
Accessibility near-zero: only 4 ARIA attributes in 4,700 LOC
Icon-only buttons have no aria-label. No role attributes on custom components. Form inputs lack associated labels. No axe-core or Playwright a11y checks in CI.
High
No client-side input validation or field-level error mapping
When the API returns HTTP 422, the form does not highlight problematic inputs or show inline error messages. Users must read the full error response in the browser console.
High
No global toast/notification system
Success and error outcomes shown as inline state resets. When users perform create → edit → delete in sequence, there is no transient confirmation that previous operations succeeded.
Medium
No rate limiting on render endpoint — unbounded component resolution
A 20-level nested component tree issues 20 synchronous DB queries per request. The frontend fans out Promise.all with no cap, potentially overwhelming slow backends.
Medium
E2E tests show high environment sensitivity
Multiple e2e tests pass locally but fail in CI, requiring 4 sequential fix cycles post-merge. Playwright has no --retries=2 configuration.
Low
No keyboard navigation support — mouse-only UI
Component insertion, tag selection, and agent association buttons require mouse clicks. No tab order or keyboard shortcuts for common actions.
Recommendations
  • Wrap all top-level routes in React ErrorBoundary; log errors to stderr and display a fallback UI with a 'Try Again' button and contact support link.
  • Add aria-label to all 12+ icon-only buttons. Add form labels and ensure input elements have accessible names. Run axe-core CLI in CI to catch regressions.
  • Add HTML5 required attributes and implement client-side validation before API calls. Map HTTP 422 field errors back to specific input elements with inline error messages.
  • Implement a toast notification library (react-hot-toast or custom) for all create/update/delete operations to provide transient feedback without state loss.
  • Add depth cap (max_depth=10, max_components=50) in render_prompt() and rate-limit the /api/prompts/{id}/render endpoint to 10 requests per minute per IP.
  • Add Playwright --retries=2 and standardize e2e seed data via seed.spec.ts. Set up a pre-merge staging e2e run matching CI environment exactly.
  • Enable keyboard navigation: add role attributes, Tab order, and onKeyDown handlers for common actions. Document keyboard shortcuts in a Help menu.

Architecture & System Design
58
F
Well-structured monolithic architecture with clear separation of concerns across frontend, backend, and database layers. However, critical resilience pattern gaps (zero timeout/retry/circuit-breaker coverage), unbounded component rendering depth, and lack of explicit transaction management create systemic fragility and scaling risks.
Findings
Critical
0% Resilience Pattern Coverage
Frontend axios client has no timeout, retry, or circuit breaker logic. Backend has no request timeouts or database query limits. A single slow query blocks all frontend requests.
Critical
Unbounded Component Rendering Depth
Recursive {{component:id}} resolution in prompt_service.py has no depth limit and triggers a database query per level. Circular references detected only at runtime via exception, not prevented upfront.
High
Single Database Creates Tight Coupling
All operations converge on one database. Cascade deletes on Prompt affect 5 related tables; any schema migration is a full redeploy. No connection pooling, idle timeouts, or read replicas.
High
No Explicit Transaction Boundaries or Error Handling
Transactions use implicit db.commit() patterns scattered across six API endpoints. Missing db.rollback() on error leaves partial state. Errors in variable substitution mid-render can corrupt the session.
High
N+1 Query Patterns and Synchronous API Design
Frontend loads tags, agents, and prompts in sequence (3+ HTTP round trips per page load). GET /prompts/{id}/versions performs linear traversal of version tree. No batching or caching.
Medium
No API Versioning or Deprecation Path
Breaking schema changes force all clients to update immediately. No feature flags, canary deployments, or gradual rollouts — all-or-nothing deployments increase blast radius.
Medium
Manual Database Migration Management
Schema changes via one-off Python scripts with no version tracking or rollback mechanism. Schema drift between local dev SQLite and production PostgreSQL.
Medium
Monolithic Backend Redeploy for All Changes
Single FastAPI application bundles Prompts, Tags, Agents, MCP server, and health checks. Any endpoint change redeploys the entire service.
Low
Frontend Tight Coupling to Backend
React SPA fetches version from /api/health on every load with no caching. No request interceptors for logging or correlation IDs.
Recommendations
  • Add HTTP request timeout (3–5s) and exponential backoff retry (max 3 attempts, 100ms base) to frontend axios client and backend HTTP calls; implement circuit breaker for /api/* endpoints.
  • Limit component rendering recursion depth to 5 levels; implement LRU cache (TTL 5 minutes) for rendered components; move render logic to read-only transaction.
  • Implement database connection pooling (min 5, max 20) and configure idle timeout; use PostgreSQL in all environments; add query timeout (500ms queries, 5s render calls).
  • Wrap all database operations in try-except-finally with explicit rollback on error; add correlation IDs to all logs; instrument render_prompt with query count and depth metrics.
  • Batch load related resources via GraphQL or composite API endpoint to reduce frontend round trips; add ETag/cache-control headers for GET requests.
  • Introduce API versioning (/api/v1/prompts); add feature flag system for gradual rollouts; implement canary deployments via Helm hooks.
  • Migrate database migrations to Alembic; add schema versioning table; test schema changes against production backup before deploy.
  • Cache frontend health check response (60s TTL); add offline mode with local storage fallback; instrument frontend with OpenTelemetry.
