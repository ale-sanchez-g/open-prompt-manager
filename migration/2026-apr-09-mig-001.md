# MIG-001: Add `agents.updated_at`

## Summary

This change adds an `updated_at` column to the `agents` table to support the agent detail API and any prompt serialization paths that load related agents.

The application code already assumes this column exists in `Agent` ORM mappings. Without the column, requests that serialize agent relationships can fail with:

```text
sqlite3.OperationalError: no such column: agents.updated_at
```

## Architecture Check

### What is sound

- The schema change is additive only. It does not drop or rewrite existing data.
- The migration backfills `updated_at` from `created_at` when available, and otherwise uses the current timestamp.
- The migration script is idempotent for normal operations: if the column already exists, it exits without changing data.
- The same script works for local SQLite and AWS PostgreSQL because it chooses `DATETIME` for SQLite and `TIMESTAMP` for PostgreSQL.

### Main architectural constraint

- This repository does not currently use a migration framework.
- `Base.metadata.create_all()` only creates missing tables. It does not alter existing tables.
- Because of that, code and database schema can drift, and application startup will not repair the drift.

### Operational requirement

- The migration must run before deploying application code that reads `agents.updated_at`.
- Do not rely on app startup to perform this change, especially in ECS where multiple tasks may start concurrently.

### Risks

- If new backend code is deployed before the migration, endpoints that load `Agent` rows can fail.
- If the migration is executed concurrently by multiple processes, there is a race between the column existence check and `ALTER TABLE`.
- If rollback is needed after code deploy, the database column can remain in place safely because the change is additive.

### Recommendation

- Treat this as a manual pre-deploy migration.
- Run it once per environment.
- Verify the column exists before shifting traffic to backend code that depends on it.

## Implementation

Migration script:

- `backend/migrations/add_agent_updated_at.py`

Behavior:

1. Connects using `DATABASE_URL`.
2. Checks whether `agents.updated_at` already exists.
3. Adds the column if missing.
4. Backfills all existing rows with `COALESCE(created_at, CURRENT_TIMESTAMP)`.

## Deployment Steps

### Local Docker

1. Build the image containing the migration script.
2. Run the migration inside the backend container.
3. Verify the schema.
4. Restart or redeploy the backend if needed.

Commands:

```bash
cd <repo-root>
docker-compose up -d --build
docker-compose exec -T backend python -m migrations.add_agent_updated_at
docker-compose exec -T backend python - <<'PY'
from app.database.base import SessionLocal
from sqlalchemy import text

db = SessionLocal()
try:
	cols = db.execute(text("PRAGMA table_info(agents)")).fetchall()
	print([c[1] for c in cols])
finally:
	db.close()
PY
```

Expected result:

- `updated_at` appears in the `agents` columns list.

### AWS ECS + RDS PostgreSQL

Run the migration in an environment where the ECS task secret-provided `DATABASE_URL` is available, or provide the same value manually.

Preferred rollout order:

1. Prepare the backend image containing the migration script.
2. Run the migration once against the target RDS database.
3. Verify the column exists and existing rows are backfilled.
4. Deploy backend tasks that depend on `agents.updated_at`.

Recommended execution options:

1. Run a one-off ECS task using the backend image and command:

```bash
python -m migrations.add_agent_updated_at
```

2. Or open a shell in a backend task/container and run the same command manually.

3. Or run it from an operator machine only if `DATABASE_URL` is explicitly pointed at the AWS RDS instance.

PostgreSQL verification query:

```sql
SELECT column_name
FROM information_schema.columns
WHERE table_name = 'agents'
ORDER BY ordinal_position;
```

Backfill verification query:

```sql
SELECT COUNT(*) AS null_updated_at_rows
FROM agents
WHERE updated_at IS NULL;
```

Expected result:

- `updated_at` exists.
- `null_updated_at_rows = 0`.

## Rollback

Application rollback:

- Safe. The added column is backward compatible with older code.

Database rollback:

- Usually unnecessary for this change.
- If absolutely required, dropping a column should be handled as a separate, explicit maintenance task.

## Validation Checklist

- Migration script is present in the deployed backend image.
- Migration executed successfully in the target environment.
- `agents.updated_at` exists.
- Existing rows have non-null `updated_at` values.
- `GET /api/prompts/` returns `200`.
- `GET /api/agents/{id}` returns `200`.

## Follow-up Architecture Recommendation

This change is safe as a one-off patch, but the underlying issue is lack of schema migration management. The next architectural step should be to introduce a real migration workflow so additive schema changes are versioned, reviewable, and applied consistently across local, Docker, and AWS environments.
